# /qa - 종합 품질 검증 워크플로우

사용자가 `/qa` 뒤에 검증 대상을 입력하면 아래 워크플로우를 실행한다.

**모든 설명과 보고서는 반드시 한국어로 작성한다.**

## 실행 규칙

- `.claude/skills/qa/` 하위의 전문가 md 파일을 역할 지시서(role prompt)로 사용한다
- 검증 대상이 명확하지 않으면 최근 변경된 파일(`git diff`, `git status`)을 기본 대상으로 삼는다
- 3명의 전문가를 **병렬로** 실행하여 속도를 확보한다
- 각 전문가는 독립된 관점으로 평가 → 결과를 종합하여 최종 보고서 작성
- 실제 버그나 중대한 보안 취약점에 우선순위를 둔다 (이론적 위험보다 실질 위험)

## 전문가 파일 매핑

| 파일 | 역할 | 책임 영역 |
|------|------|-----------|
| `qa/code-reviewer.md` | Staff Engineer 코드 리뷰어 | 정확성, 가독성, 아키텍처, 성능 |
| `qa/security-auditor.md` | Security Engineer | 입력 검증, 인증/인가, 암호화, 취약점 |
| `qa/test-engineer.md` | QA Engineer | 테스트 전략, 커버리지, 테스트 작성 |

## 워크플로우

### Phase 1: 검증 범위 확정
사용자가 명시한 대상을 파악한다:
- **파일/디렉토리 지정**: 해당 경로만 검증
- **"최근 변경사항"** 또는 명시 없음: `git diff HEAD` 또는 `git status`로 변경 파일 수집
- **PR/브랜치**: `git diff main...HEAD`로 diff 수집
- **특정 기능**: 관련 파일을 `Explore` 에이전트로 식별

검증 범위가 너무 넓으면 (50+ 파일) 사용자에게 좁힐지 확인한다.

### Phase 2: 전문가 병렬 실행

3개의 `general-purpose` 서브에이전트를 **한 번의 메시지**에서 병렬로 실행한다. 각 에이전트에게 전달할 프롬프트 구조:

```
다음 역할 지시서를 읽고, 역할에 맞춰 아래 대상을 평가해줘:

역할 지시서:
<.claude/skills/qa/{전문가}.md 파일 내용 전체>

검증 대상:
- 파일 경로 목록: ...
- 변경 범위 (git diff 요약): ...
- 프로젝트 컨텍스트: ...

응답 형식:
- 심각도: critical / high / medium / low / info
- 각 이슈는: [심각도] 위치(file:line) — 문제 설명 — 권장 수정안
- 종합 요약: 현재 코드의 주요 강점과 약점 3가지씩
- 마지막에 '즉시 수정 필요' / '머지 가능' / '추가 작업 권장' 중 판정

대상의 범위를 벗어난 사소한 지적은 생략한다.
```

각 에이전트는 서로의 결과를 보지 못하며, 독립된 관점으로 평가한다.

### Phase 3: 결과 종합 및 보고

세 전문가의 응답을 수신한 후 다음 형식으로 최종 보고서를 작성한다:

```
# QA 검증 보고서

## 종합 판정
[통과 / 조건부 통과 / 재작업 필요]

## 핵심 요약 (3줄)
- ...
- ...
- ...

## 영역별 결과

### 🔎 코드 리뷰 (code-reviewer)
- Critical: N건
- High: N건
- [주요 이슈 3개 인라인]

### 🔐 보안 감사 (security-auditor)
- Critical: N건
- High: N건
- [주요 이슈 3개 인라인]

### 🧪 테스트 품질 (test-engineer)
- 커버리지 평가
- 테스트 누락 영역
- [주요 권장사항 3개]

## 즉시 조치 필요 (Critical/High)
1. [영역] 파일:라인 — 문제 — 수정안
2. ...

## 중기 개선 권장 (Medium)
- ...

## 각 전문가 전체 응답
<접기/펼치기 형태로 각 에이전트 원본 보고서 첨부>
```

### Phase 4: 후속 조치 제안

사용자에게 선택지를 제공한다:
- `즉시 수정` — Critical/High 이슈를 자동으로 수정 착수 (senior-code-architect 에이전트 활용)
- `부분 수정` — 특정 이슈만 선택하여 수정
- `리포트만` — 보고서만 저장하고 종료 (파일명: `qa-report-YYYYMMDD-HHMM.md`)
- `테스트 추가` — test-engineer 권장에 따라 테스트 코드 작성

## Serena MCP 활용 (있는 경우)

- `serena_find_references`로 이슈가 있는 함수/클래스의 영향 범위 확인
- `serena_get_diagnostics`로 LSP 수준의 에러/경고를 교차 검증
- Serena 없이도 동작 가능 — Grep/Read로 폴백

## 사용 예시

```
/qa
/qa 최근 변경사항 검토해줘
/qa packages/signaling-server/src/diagnosis-api.ts
/qa PR 머지 전 최종 점검
/qa 보안 위주로만 봐줘   ← 이 경우 security-auditor만 실행
/qa 테스트 커버리지 확인   ← 이 경우 test-engineer만 실행
```

특정 전문가만 요청하는 경우 해당 전문가 1명만 실행하고 보고서도 1개 영역으로 축약한다.

## 주의사항

- 실제 코드를 수정하지는 않는다 (Phase 4에서 사용자가 요청한 경우만)
- 전문가 md 파일의 원문을 그대로 역할 지시서로 사용하되, 중복 지시는 피한다
- 빌드/타입 체크가 가능한 경우 사전 실행하여 명백한 에러를 먼저 걸러낸다
- 이슈 보고 시 파일 경로는 `[파일명.ext:라인번호](경로#L라인)` 형식의 마크다운 링크를 사용한다
